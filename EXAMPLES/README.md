# STM32 Examples

Bare-metal example projects for several STM32 microcontrollers, generated by
the [`stm32codegen`](https://github.com/a5021/stm32codegen) tool. Each example
blinks the board's on-board LED using only CMSIS (no HAL, no external
libraries) and is wired for three toolchains out of the box: GNU Make
(arm-none-eabi-gcc), Keil MDK-ARM, and SEGGER Embedded Studio (plus Ozone /
J-Link for flashing and debugging).

---

## What's in this folder

| Generator script      | Target MCU        | Board / LED pin        | Core / HCLK | Notes                                   |
|-----------------------|-------------------|------------------------|-------------|-----------------------------------------|
| `f103c8_demo.sh`      | STM32F103C8       | Blue Pill, **PC13**     | Cortex-M3, 8 MHz HSI | The reference example; documented in detail below. |
| `f030f4_demo.sh`      | STM32F030F4       | STM32F0 Discovery, **PA4** | Cortex-M0, 8 MHz HSI | `gpio.h` is generated by `stm32cgen` (`GPIOA GPIOB GPIOF`), not hand-written. |
| `g031f8_demo.sh`      | STM32G031F8       | NUCLEO-G031, **PA5**    | Cortex-M0+, 16 MHz HSI | G0 needs flash wait states above 48 MHz. |
| `f401cc_demo.sh`      | STM32F401CC       | BlackPill, **PC13**     | Cortex-M4F, 100 MHz HSE | Overclocked (max rated 84 MHz). HSE 25 MHz crystal, PLL 25/15×120/2. |
| `f303vc_demo.sh`      | STM32F303VC       | STM32F3 Discovery, **PE8–PE15** | Cortex-M4F, 72 MHz HSE | 8 on-board LEDs; LED chase via SysTick. HSE 8 MHz from ST-LINK MCO, PLL ×9. |
| `f407vg_demo.sh`      | STM32F407VG       | STM32F4 Discovery, **PD12–PD15** | Cortex-M4F, 168 MHz HSE | 4 on-board LEDs (active LOW); LED chase via SysTick. HSE 8 MHz crystal, PLL ×21. |

The generated output folders (`f103c8_demo/`, `f030f4_demo/`, `g031f8_demo/`,
`f401cc_demo/`, `f303vc_demo/`, `f407vg_demo/`) are build artifacts produced by running the
matching script; they are **not** checked into the repository. Re-run a script
to regenerate.

## How the generators work

Each `*.sh` script is a self-contained project generator. Running it:

1. Creates the directory tree (`inc/`, `src/`, `MDK-ARM/`).
2. Generates the CMSIS-based configuration headers with `stm32cgen`:
   - `inc/main.h` — top-level config (clock, SysTick, init order) and the
     `init()` sequence (`init_rcc` → `init_gpio` → `init_systick`).
   - `inc/rcc.h` — `init_rcc()` and the RCC clock-enable bits.
   - `inc/gpio.h` — `init_gpio()` and the GPIO macros. For the F1
     (`f103c8_demo`) this file is hand-written in the
     [BluePill_Project_Generator](https://github.com/a5021/BluePill_Project_Generator)
     style; for F0/F4/G0 it is emitted by `stm32cgen`.
3. Downloads the vendor CMSIS files (device header, startup, system init) and
   the SVD description from upstream sources (cached for re-runs).
4. Writes the application source `src/main.c`, the linker script, the Makefile,
   and the debugger/IDE configurations.
5. Builds the project with `make` to verify everything is consistent.

All downloaded files are cached — re-running the script skips what already
exists and only rebuilds.

### Prerequisites

- `bash` (native Linux/macOS, Cygwin, or WSL on Windows)
- `arm-none-eabi-gcc`, `make`, `curl`
- `stm32cgen` available via the `PY_GEN` environment variable (defaults to the
  repository root, i.e. one level above `EXAMPLES/`)

```sh
# from the EXAMPLES/ directory
bash ./f103c8_demo.sh
```

## How the demo works (f103c8 reference)

### Clock and SysTick

`main.h` sets `HCLK = 8` MHz (HSI, no PLL). `init_systick()` configures the
Cortex-M SysTick to fire every 1 ms:

```c
SysTick->LOAD = HCLK * 1000 / (8 - SYSTICK_CLOCK_SOURCE * 7) - 1;
```

With `SYSTICK_CLOCK_SOURCE = 0` the SysTick runs from `HCLK / 8` (1 MHz),
giving a 1 ms period. `SYSTICK_IRQ_ENABLE` selects between polling mode
(the default) and interrupt mode.

### GPIO — PC13 LED

The on-board LED is connected to **PC13** (active-low: writing the pin high
turns the LED off, low turns it on). `gpio.h` configures it with:

```c
#define PORT_C_CONFIG  PIN_CFG(13, O_2MHZ)   /* PC13: output, push-pull, 2 MHz */
#define PORT_C_STATE   PIN_HIGH(13)          /* PC13 HIGH = LED off            */
#define IOPC_EN        (PORT_C_CONFIG != 0)
```

`init_gpio()` first writes the idle state to `GPIOC->BSRR`, then writes the
full 64-bit `CRL+CRH` configuration through `*(uint64_t*)GPIOC_BASE`. Because
`IOPC_EN` is non-zero, `init_rcc()` enables `RCC->APB2ENR.IOPCEN`, which clocks
the GPIOC peripheral (on STM32F1 each GPIO port is gated by its own APB2 bit).

### Main loop

`src/main.c` runs the standard `for (init(); process(); idle());` loop. In
polling mode `idle()` calls `process_systick_event()`, which waits for the 1 ms
SysTick flag, increments a 64-bit uptime counter, and toggles PC13 based on
bit 9 of the counter — producing a visible blink with a period of
2^10 = 1024 ticks (~1 second).

In interrupt mode (`SYSTICK_IRQ_ENABLE = YES`) the same toggle logic runs inside
`SysTick_Handler()` and the `idle()` stub becomes empty.

## Building

### GNU Make (arm-none-eabi-gcc)

```sh
make            # minimal build -> _build/Project.elf (+ .hex, .bin)
make debug      # debug build (-Og -g3 -gdwarf)
make size       # print ELF section sizes
make program    # flash via ST-Link
make jprogram   # flash via J-Link
make clean      # remove build output
```

### Keil MDK-ARM

Open `MDK-ARM/Project.uvprojx` in µVision, or build headlessly:

```sh
UV4.exe -b MDK-ARM/Project.uvprojx -t "Debug"
```

The MDK project references the SVD from the demo root, so the peripheral
windows are populated during debugging without installing any device pack.

### SEGGER Embedded Studio

Open `project.emProject`, or build headlessly with `emBuild`:

```sh
emBuild.exe -config Debug project.emProject
```

SES uses its own built-in startup files; the external `startup_stm32????.s`
is only consumed by the MDK project.

## Flashing and debugging

Connect a J-Link (or ST-Link) to the board: SWDIO → PA13, SWCLK → PA14,
GND → GND, 3.3V → 3.3V (pinout varies by board — check the silk screen).

- **Ozone** (J-Link): open `project.jdebug`; it loads `_build/Project.elf`
  and the shared SVD.
- **J-Link Commander** flash example (LED blink verified on Blue Pill):
  ```sh
  JLink.exe -device STM32F103CB -if SWD -speed 4000 \
            -CommanderScript flash.jlink
  ```
  where `flash.jlink` contains `loadfile`, `r`, `g`.
- **MDK / SES**: use the integrated flash + debug buttons.

## Configuration knobs

The main tunables live at the top of `inc/main.h` (generated by `stm32cgen`):

| Macro                 | Default (f103c8) | Meaning                                  |
|-----------------------|------------------|------------------------------------------|
| `HCLK`                | `8`              | System clock in MHz (8–72 for F1, step 4)|
| `SYSTICK_CLOCK_SOURCE`| `0`              | 0 = HCLK/8, 1 = HCLK                     |
| `SYSTICK_ENABLE`      | `YES`            | enable SysTick                           |
| `SYSTICK_IRQ_ENABLE`  | `NO`             | IRQ vs polling mode                      |

To change the blinked pin or add peripherals, edit `PORT_C_CONFIG` in
`inc/gpio.h` (or extend it with `PORT_A_CONFIG` / `PORT_B_CONFIG` using the
same `PIN_CFG` macros) and the matching `IOPx_EN` enable bits. For the
F0/F4/G0 demos, `gpio.h` is regenerated by the script, so adjust the
`stm32cgen` GPIO invocation in the `.sh` script instead.

## F303VC Discovery demo

### Board

STM32F303VC Discovery (Device ID 0x422, 256 KB Flash, 40 KB SRAM1 + 8 KB SRAM2).
8 user LEDs on **PE8–PE15** (active-high). The ST-LINK/V2 on-board provides an
8 MHz HSE clock on the MCO pin (PA8) which is routed to the MCU HSE input.

### Clock

`main.h` sets `HCLK = 72` MHz. `wait_for_clock_stable()` in `rcc.h` configures:

- HSE = 8 MHz (from ST-LINK MCO)
- PLL input = HSE / PREDIV1 = 8 MHz
- PLL multiplier = ×9 → PLL output = 72 MHz
- AHB prescaler = ÷1 → HCLK = 72 MHz
- APB1 prescaler = ÷2 → PCLK1 = 36 MHz
- APB2 prescaler = ÷1 → PCLK2 = 72 MHz

Flash latency is set to 2 wait states with prefetch enabled for 72 MHz.

### GPIO — PE8–PE15 LEDs

All 8 LED pins are configured as push-pull outputs:

```c
#define GPIOE_MODE (                              \
  !0 * PIN_MODE(8,  PIN_MODE_OUTPUT) /* PE8  -- LD4  Blue    */ | \
  !0 * PIN_MODE(9,  PIN_MODE_OUTPUT) /* PE9  -- LD3  Red     */ | \
  ...                                                                  \
  !0 * PIN_MODE(15, PIN_MODE_OUTPUT) /* PE15 -- LD6  Green   */   \
)
```

`init_gpio()` writes `GPIOE_MODER` (and other registers) directly. The
`GPIOE_EN` flag is auto-derived: `init_rcc()` sets `RCC->APB2ENR` bit 21 to
clock GPIOE.

### LED chase

SysTick runs at 1 kHz (HCLK/8 = 9 MHz, LOAD = 8999). In polling mode,
`idle()` checks `SysTick->CTRL.COUNTFLAG` each iteration and increments a
64-bit `system_uptime` counter:

```c
uint64_t t = ++*uptime();
uint32_t led_bit = (t >> 9) & 7;         /* LED index 0–7, cycles every 512 ms */
uint32_t led_mask = 1UL << (8 + led_bit); /* PE8–PE15                         */

if (t & (1 << 8)) {                       /* on/off phase every 256 ms        */
  GPIOE->BRR  = 0xFF00;                   /* turn off all PE8–PE15            */
  GPIOE->BSRR = led_mask;                 /* turn on current LED              */
} else {
  GPIOE->BRR = led_mask;                  /* turn off current LED             */
}
```

Bit [8] toggles on/off (256 ms period), bits [11:9] select the active LED.
Result: one LED lights at a time, cycling PE8 → PE9 → … → PE15 → PE8.

### Linker script

GCC linker script places `_estack = 0x2000A000` (top of SRAM1, 40 KB) — matching
the official ST linker scripts which do not place stack in SRAM2. An assertion
guards against overflow:

```ld
ASSERT(_estack - _end >= _Min_Stack_Size,
  "RAM overflow: .data + .bss + stack exceeds 40 KB")
```

The Keil auto-generated scatter file behaves differently: it places stack + BSS
in CCM (0x10000000, 8 KB). Keil defaults to placing ZI/stack in the last IRAM
region, and CCM (`IRAM2`) is configured as the last region in the project
settings. Both builds work correctly. This is specific to the F303 — it is the
only example with CCM mapped at a separate address (0x10000000).

### Building and flashing

```sh
bash ./f303vc_demo.sh        # generate + build (debug target)
# or from the generated directory:
cd f303vc_demo
make debug                   # debug build (-Og -g3 -gdwarf)
make program                 # flash via ST-Link CLI
```

ST-Link CLI requires the `-c SWD ... UR` (under-reset) option to connect
while the MCU is running:

```sh
ST-LINK_CLI.exe -c SWD SWCLK=4000 UR -P _build/Project.hex -Rst -Run
```

## F407VG Discovery demo

### Board

STM32F4 Discovery (MB997C, STM32F407VGT6). 4 user LEDs on **PD12–PD15**
(active-low: pin LOW = LED ON). 8 MHz HSE crystal. On-board ST-LINK does
**not** route MCO to the target.

### Clock

`main.h` sets `HCLK = 168` MHz. `wait_for_clock_stable()` in `rcc.h`
configures:

- HSE = 8 MHz (crystal)
- PLLM = 8, PLLN = 336, PLLP = 2 → 168 MHz
- PLLQ = 7 → USB clock = 48 MHz
- AHB prescaler = ÷1 → HCLK = 168 MHz
- APB1 prescaler = ÷4 → PCLK1 = 42 MHz
- APB2 prescaler = ÷2 → PCLK2 = 84 MHz

PWR voltage scaling set to VOS1. Flash latency = 5 wait states with
prefetch + I-cache + D-cache.

### GPIO — PD12–PD15 LEDs

```c
#define GPIOD_MODE (                              \
  !0 * PIN_MODE(12, PIN_MODE_OUTPUT) /* PD12 -- LD3  Green   */ | \
  !0 * PIN_MODE(13, PIN_MODE_OUTPUT) /* PD13 -- LD4  Orange  */ | \
  !0 * PIN_MODE(14, PIN_MODE_OUTPUT) /* PD14 -- LD5  Red     */ | \
  !0 * PIN_MODE(15, PIN_MODE_OUTPUT) /* PD15 -- LD6  Blue    */   \
)
```

### LED chase

LEDs are **active LOW** (pin LOW = ON). The chase logic uses `BSRR` only
(F4 has no BRR register):

```c
uint64_t t = ++*uptime();
uint32_t led_bit = (t >> 9) & 3;            /* LED index 0–3          */
uint32_t led_mask = 1UL << (12 + led_bit);  /* PD12–PD15              */

if (t & (1 << 8)) {
  GPIOD->BSRR = 0xF000;                     /* all HIGH = off         */
  GPIOD->BSRR = 1UL << (16 + 12 + led_bit); /* current LOW = on      */
} else {
  GPIOD->BSRR = led_mask;                   /* current HIGH = off    */
}
```

### Linker script

GCC linker: `_estack` at top of SRAM1 (0x2001C000, 112 KB). CCM (64 KB at
0x10000000) available for `__attribute__((section(".ccmram")))` data.

Keil: IRAM = 112 KB @ 0x20000000, IRAM2 = 64 KB CCM @ 0x10000000.

### Building and flashing

```sh
bash ./f407vg_demo.sh        # generate + build (debug target)
# or from the generated directory:
cd f407vg_demo
make debug                   # debug build (-Og -g3 -gdwarf)
make program                 # flash via ST-Link CLI
```
