# STM32 Examples

Bare-metal example projects for several STM32 microcontrollers, generated by
the [`stm32codegen`](https://github.com/a5021/stm32codegen) tool. Each example
blinks the board's on-board LED using only CMSIS (no HAL, no external
libraries) and is wired for three toolchains out of the box: GNU Make
(arm-none-eabi-gcc), Keil MDK-ARM, and SEGGER Embedded Studio (plus Ozone /
J-Link for flashing and debugging).

---

## What's in this folder

| Generator script      | Target MCU        | Board / LED pin        | Core / HCLK | Notes                                   |
|-----------------------|-------------------|------------------------|-------------|-----------------------------------------|
| `f103c8_demo.sh`      | STM32F103C8       | Blue Pill, **PC13**     | Cortex-M3, 8 MHz HSI | The reference example; documented in detail below. |
| `f030f4_demo.sh`      | STM32F030F4       | STM32F0 Discovery, **PA4** | Cortex-M0, 8 MHz HSI | `gpio.h` is generated by `stm32cgen` (`GPIOA GPIOB GPIOF`), not hand-written. |
| `g031f8_demo.sh`      | STM32G031F8       | NUCLEO-G031, **PA5**    | Cortex-M0+, 16 MHz HSI | G0 needs flash wait states above 48 MHz. |

The generated output folders (`f103c8_demo/`, `f030f4_demo/`, `g031f8_demo/`)
are build artifacts produced by running the matching script; they are **not**
checked into the repository. Re-run a script to regenerate.

> A legacy `f103c8t6_demo/` folder may also be present. It is an older generated
> output and has no corresponding generator script in this folder — treat it as
> historical.

## How the generators work

Each `*.sh` script is a self-contained project generator. Running it:

1. Creates the directory tree (`inc/`, `src/`, `MDK-ARM/`).
2. Generates the CMSIS-based configuration headers with `stm32cgen`:
   - `inc/main.h` — top-level config (clock, SysTick, init order) and the
     `init()` sequence (`init_rcc` → `init_gpio` → `init_systick`).
   - `inc/rcc.h` — `init_rcc()` and the RCC clock-enable bits.
   - `inc/gpio.h` — `init_gpio()` and the GPIO macros. For the F1
     (`f103c8_demo`) this file is hand-written in the
     [BluePill_Project_Generator](https://github.com/a5021/BluePill_Project_Generator)
     style; for F0/G0 it is emitted by `stm32cgen`.
3. Downloads the vendor CMSIS files (device header, startup, system init) and
   the SVD description from upstream sources (cached for re-runs).
4. Writes the application source `src/main.c`, the linker script, the Makefile,
   and the debugger/IDE configurations.
5. Builds the project with `make` to verify everything is consistent.

All downloaded files are cached — re-running the script skips what already
exists and only rebuilds.

### Prerequisites

- `bash` (native Linux/macOS, Cygwin, or WSL on Windows)
- `arm-none-eabi-gcc`, `make`, `curl`, `base64`, `xz`
- `stm32cgen` available via the `PY_GEN` environment variable (defaults to the
  repository root, i.e. one level above `EXAMPLES/`)

```sh
# from the EXAMPLES/ directory
bash ./f103c8_demo.sh
```

```
├── Makefile                  # GNU make build (arm-none-eabi-gcc)

├── project.jdebug            # Ozone debugger script
├── project.jflash            # J-Link flash script
├── inc/
│   ├── main.h                # configuration + init() sequence
│   ├── rcc.h                 # init_rcc() / RCC clock enables
│   ├── gpio.h                # init_gpio() + GPIO macros
│   ├── stm32????.h           # CMSIS device header (downloaded)
│   ├── system_stm32?xx.h
│   └── ...                   # other CMSIS/core headers
├── src/
│   ├── main.c                # application: init/process/idle loop, LED toggle
│   ├── system_stm32f1xx.c    # SystemInit() stub (HSI default)
│   └── ...
├── MDK-ARM/
│   ├── Project.uvprojx       # Keil MDK-ARM project
│   └── startup_stm32????.s   # ARM/Keil assembly startup (for MDK)
└── _build/                   # make output: Project.elf / .hex / .bin / .map
```

> The SVD file lives at the demo root so it can be shared by all three
> debugger configs: MDK references it via `<SFDFile>..\STM32F103xx.svd</SFDFile>`,
> Ozone via `$(ProjectDir)/STM32F103xx.svd`, and SES via
> `debug_svd_file="$(ProjectDir)/STM32F103xx.svd"`.

## How the demo works (f103c8 reference)

### Clock and SysTick

`main.h` sets `HCLK = 8` MHz (HSI, no PLL). `init_systick()` configures the
Cortex-M SysTick to fire every 1 ms:

```c
SysTick->LOAD = HCLK * 1000 / (8 - SYSTICK_CLOCK_SOURCE * 7) - 1;
```

With `SYSTICK_CLOCK_SOURCE = 0` the SysTick runs from `HCLK / 8` (1 MHz),
giving a 1 ms period. `SYSTICK_IRQ_ENABLE` selects between polling mode
(the default) and interrupt mode.

### GPIO — PC13 LED

The on-board LED is connected to **PC13** (active-low: writing the pin high
turns the LED off, low turns it on). `gpio.h` configures it with:

```c
#define PORT_C_CONFIG  PIN_CFG(13, O_2MHZ)   /* PC13: output, push-pull, 2 MHz */
#define PORT_C_STATE   PIN_HIGH(13)          /* PC13 HIGH = LED off            */
#define IOPC_EN        (PORT_C_CONFIG != 0)
```

`init_gpio()` first writes the idle state to `GPIOC->BSRR`, then writes the
full 64-bit `CRL+CRH` configuration through `*(uint64_t*)GPIOC_BASE`. Because
`IOPC_EN` is non-zero, `init_rcc()` enables `RCC->APB2ENR.IOPCEN`, which clocks
the GPIOC peripheral (on STM32F1 each GPIO port is gated by its own APB2 bit).

### Main loop

`src/main.c` runs the standard `for (init(); process(); idle());` loop. In
polling mode `idle()` calls `process_systick_event()`, which waits for the 1 ms
SysTick flag, increments a 64-bit uptime counter, and toggles PC13 based on
bit 9 of the counter — producing a visible blink with a period of
2^10 = 1024 ticks (~1 second).

In interrupt mode (`SYSTICK_IRQ_ENABLE = YES`) the same toggle logic runs inside
`SysTick_Handler()` and the `idle()` stub becomes empty.

## Building

### GNU Make (arm-none-eabi-gcc)

```sh
make            # minimal build -> _build/Project.elf
make debug      # debug build (-Og -g3, SWD enabled)
make full       # all peripherals enabled
make program    # flash via ST-Link (make flash target)
```

### Keil MDK-ARM

Open `MDK-ARM/Project.uvprojx` in µVision, or build headlessly:

```sh
UV4.exe -b MDK-ARM/Project.uvprojx -t "Debug"
```

The MDK project references the SVD from the demo root, so the peripheral
windows are populated during debugging without installing any device pack.

### SEGGER Embedded Studio

Open `project.emProject`, or build headlessly with `emBuild`:

```sh
emBuild.exe -config Debug project.emProject
```

SES uses its own built-in startup files; the external `startup_stm32????.s`
is only consumed by the MDK project.

## Flashing and debugging

Connect a J-Link (or ST-Link) to the board: SWDIO → PA13, SWCLK → PA14,
GND → GND, 3.3V → 3.3V (pinout varies by board — check the silk screen).

- **Ozone** (J-Link): open `project.jdebug`; it loads `_build/Project.elf`
  and the shared SVD.
- **J-Link Commander** flash example (LED blink verified on Blue Pill):
  ```sh
  JLink.exe -device STM32F103CB -if SWD -speed 4000 \
            -CommanderScript flash.jlink
  ```
  where `flash.jlink` contains `loadfile`, `r`, `g`.
- **MDK / SES**: use the integrated flash + debug buttons.

## Configuration knobs

The main tunables live at the top of `inc/main.h` (generated by `stm32cgen`):

| Macro                 | Default (f103c8) | Meaning                                  |
|-----------------------|------------------|------------------------------------------|
| `HCLK`                | `8`              | System clock in MHz (8–72 for F1, step 4)|
| `SYSTICK_CLOCK_SOURCE`| `0`              | 0 = HCLK/8, 1 = HCLK                     |
| `SYSTICK_ENABLE`      | `YES`            | enable SysTick                           |
| `SYSTICK_IRQ_ENABLE`  | `NO`             | IRQ vs polling mode                      |

To change the blinked pin or add peripherals, edit `PORT_C_CONFIG` in
`inc/gpio.h` (or extend it with `PORT_A_CONFIG` / `PORT_B_CONFIG` using the
same `PIN_CFG` macros) and the matching `IOPx_EN` enable bits. For the
F0/G0 demos, `gpio.h` is regenerated by the script, so adjust the
`stm32cgen` GPIO invocation in the `.sh` script instead.
